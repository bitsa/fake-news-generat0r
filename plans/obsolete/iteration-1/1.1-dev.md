## Implementation plan

Steps are ordered so that imports resolve cleanly (sources → ORM models →
migration → tests). Do not skip steps; each one must compile / pass before
the next is started.

1. **Create `app/sources.py`.** Define `Source(StrEnum)` with members in
   exact order `NYT`, `NPR`, `GUARDIAN`. Define `FEED_URLS: dict[Source, str]`
   with the three URLs from `contracts.md` verbatim. No other exports.

2. **Create `app/models/__init__.py` with `Base`.**
   - `from sqlalchemy.orm import DeclarativeBase`
   - `class Base(DeclarativeBase): pass`
   - Re-export `Article`, `ArticleFake`, `ChatMessage` from their modules.
   - Use `__all__` so the public surface is explicit.

3. **Create `app/models/article.py`.** Use SQLAlchemy 2.0
   `Mapped[...] / mapped_column(...)` style. Notes:
   - `id`: `mapped_column(sa.Integer, primary_key=True, autoincrement=True)`
     (SERIAL).
   - `source`: `mapped_column(sa.Enum(Source, name="source_type",
     values_callable=lambda e: [m.value for m in e]))` — the
     `values_callable` ensures the Postgres enum labels are the `.value`
     strings (`"NYT"`, `"NPR"`, `"Guardian"`), not the Python member names.
   - `title`, `description`: `Mapped[str]` with `sa.Text`, `nullable=False`.
   - `url`: `Mapped[str]` with `sa.Text`, `nullable=False`, `unique=True`.
   - `published_at`: `Mapped[datetime | None]` with
     `sa.DateTime(timezone=True)`, `nullable=True`.
   - `content_hash`: `Mapped[str]` with `sa.CHAR(64)`, `nullable=False`,
     `unique=True`.
   - `created_at`: `Mapped[datetime]` with `sa.DateTime(timezone=True)`,
     `nullable=False`, `server_default=sa.func.now()`.
   - No relationships are required for 1.1 (none of the downstream tasks in
     this iteration's surface depend on ORM-level back-refs from `Article`).
     If `Article.fake` / `Article.messages` relationships are added later
     they belong to whichever task first needs them.

4. **Create `app/models/article_fake.py`.** Notes:
   - `article_id`: `mapped_column(sa.Integer,
     sa.ForeignKey("articles.id", ondelete="CASCADE"), primary_key=True)`.
     The single `mapped_column` is **simultaneously** the PK and the FK —
     this is what enforces 1:1 by construction (ADR-5; spec
     `article_fakes` criterion 2).
   - `fake_title`, `fake_description`: `sa.Text`, `nullable=False`.
   - `model`: `sa.String(100)`, `nullable=False`.
   - `temperature`: `sa.Double` (or `sa.Float(asdecimal=False)` if `Double`
     unavailable in the installed SQLAlchemy version — match the
     contracts.md `DOUBLE PRECISION`), `nullable=False`.
   - `created_at`: `sa.DateTime(timezone=True)`, `nullable=False`,
     `server_default=sa.func.now()`.

5. **Create `app/models/chat_message.py`.** Notes:
   - `id`: SERIAL PK.
   - `article_id`: `sa.Integer`, FK to `articles.id` with
     `ondelete="CASCADE"`, `nullable=False`. (Distinct from
     `article_fakes` — here `article_id` is **not** the PK and is
     `nullable=False` rather than implicit-not-null-via-PK.)
   - `role`: `sa.String(20)`, `nullable=False`, with a table-level
     `CheckConstraint("role IN ('user', 'assistant')",
     name="ck_chat_messages_role")` (declared via `__table_args__`).
   - `content`: `sa.Text`, `nullable=False`.
   - `created_at`: same shape as the other tables.
   - **No** `user_id` column (ADR-11).

6. **Create the genesis migration**
   `backend/migrations/versions/<rev>_initial_schema.py`. Use
   `alembic revision -m "initial schema"` to generate the skeleton (this
   produces a fresh 12-char revision id), then **replace the body by hand**
   per ADR-3 — do not run `--autogenerate`.
   - Set `down_revision = None`.
   - Import `from app.sources import Source` at module top — the migration
     must reference `Source` so the labels cannot drift (ADR-16; spec
     criterion 5 of "Source StrEnum and FEED_URLS").
   - Build the enum:

     ```python
     source_type = postgresql.ENUM(
         *[s.value for s in Source],
         name="source_type",
         create_type=False,
     )
     ```

     `create_type=False` prevents SQLAlchemy from auto-creating the type
     when it appears in `create_table` — explicit lifecycle is owned by
     `upgrade` / `downgrade`.
   - `upgrade()`:
     1. `source_type.create(op.get_bind(), checkfirst=False)`.
     2. `op.create_table("articles", ...)` with columns and the
        `UniqueConstraint`s on `url` and `content_hash`. The `source`
        column must reference the `source_type` enum object created above
        (so SQLAlchemy emits `source source_type` rather than re-creating
        the type).
     3. `op.create_index("ix_articles_source", "articles", ["source"])`.
     4. `op.create_index("ix_articles_published_at", "articles",
        [sa.text("published_at DESC NULLS LAST")])` — expression index
        per `contracts.md`. Do **not** create an additional non-unique
        index on `content_hash` (the unique constraint already provides
        the lookup index — spec `articles` criterion 3).
     5. `op.create_table("article_fakes", ...)` with `article_id` as
        `INTEGER PRIMARY KEY REFERENCES articles(id) ON DELETE CASCADE`.
        No additional indexes (spec `article_fakes` criterion 7).
     6. `op.create_table("chat_messages", ...)` with the `role` CHECK
        constraint and the FK to `articles(id) ON DELETE CASCADE`.
     7. `op.create_index("ix_chat_messages_article_id", "chat_messages",
        ["article_id"])`.
   - `downgrade()` reverses in opposite order:
     1. `op.drop_index("ix_chat_messages_article_id", table_name="chat_messages")`.
     2. `op.drop_table("chat_messages")`.
     3. `op.drop_table("article_fakes")`. (Its PK index is dropped with
        the table; no separate drop_index needed.)
     4. `op.drop_index("ix_articles_published_at", table_name="articles")`.
     5. `op.drop_index("ix_articles_source", table_name="articles")`.
     6. `op.drop_table("articles")`.
     7. `source_type.drop(op.get_bind(), checkfirst=False)` — the
        `DROP TYPE source_type` required by spec "Reversibility" #1.

7. **Verify locally before declaring done.** Stand up an empty Postgres 16
   (`pgvector/pgvector:pg16` is fine — pgvector extension stays disabled
   per ADR-12), then from `backend/`:
   - `alembic upgrade head` succeeds; inspect with `psql \d` that the
     three tables and `source_type` enum match `contracts.md`.
   - `alembic downgrade base` leaves the database empty (no leftover
     types, tables, indexes — spec "Reversibility" #2).
   - `alembic upgrade head` after the downgrade succeeds (spec
     "Reversibility" #3).

8. **Run formatters and linter.** From `backend/`: `black .` and
   `ruff check .` per `conventions.md`. Both must pass with zero
   findings before commit.

## Unit tests required

Unit tests are inline with this task (Dev responsibility per
[plans/plan.md](../../plans/plan.md)). They cover behaviors that do **not**
require a live Postgres — schema-level acceptance criteria (enum violation
rejected, cascade delete observed, etc.) are integration-test territory and
belong to the QA pass for 1.1.

Tests live under `backend/tests/models/` and use `pytest-asyncio` with
`asyncio_mode = "auto"` per `conventions.md`.

### `tests/models/test_sources.py`

- `test_source_members_exact` — `set(Source) == {Source.NYT, Source.NPR,
  Source.GUARDIAN}` and `len(Source) == 3`.
- `test_source_member_values` — `.value` of each member equals
  `"NYT" / "NPR" / "Guardian"` exactly (capitalization sensitive).
- `test_source_is_strenum` — `issubclass(Source, str)` and a `Source`
  member is interchangeable with its string value (`Source.NYT == "NYT"`).
- `test_source_declaration_order` — `list(Source)` is
  `[Source.NYT, Source.NPR, Source.GUARDIAN]` in that order
  (the migration's enum label order depends on this — spec assumption #4).
- `test_feed_urls_keys_match_source` — `set(FEED_URLS) == set(Source)`.
- `test_feed_urls_values` — each URL is the exact string from
  `contracts.md`; assert non-empty and `.startswith("https://")`.
- `test_feed_urls_type` — `FEED_URLS` annotation is `dict[Source, str]`
  (read via `typing.get_type_hints`, or via the module-level `__annotations__`).

### `tests/models/test_orm_models.py`

- `test_tablenames` — `Article.__tablename__ == "articles"`,
  `ArticleFake.__tablename__ == "article_fakes"`,
  `ChatMessage.__tablename__ == "chat_messages"`.
- `test_models_share_base` — all three are subclasses of `app.models.Base`
  and registered on `Base.metadata.tables`.
- `test_metadata_table_count` — `set(Base.metadata.tables) ==
  {"articles", "article_fakes", "chat_messages"}` — guards against an
  accidental fourth table (e.g. `sources`, `article_versions`,
  `article_embeddings`) being introduced.
- `test_article_source_column_uses_source_enum` — inspect
  `Article.__table__.c.source.type`, assert it is a `sa.Enum` whose
  `enum_class is Source` and whose `name == "source_type"`. Assert that
  the enum's persisted labels equal `[m.value for m in Source]` (covers
  spec ORM criterion 3 and the values_callable wiring).
- `test_article_fake_pk_is_fk` — inspect
  `ArticleFake.__table__.c.article_id`: `primary_key is True`, exactly
  one `ForeignKey` whose `column.table.name == "articles"`,
  `column.name == "id"`, and whose `ondelete == "CASCADE"`.
- `test_article_fake_no_extra_indexes` — `len(ArticleFake.__table__.indexes)`
  is 0 (the implicit PK index is not in `.indexes`); guards spec
  `article_fakes` criterion 7.
- `test_chat_messages_role_check_constraint` — find a `CheckConstraint`
  on `ChatMessage.__table__.constraints` whose `sqltext` mentions both
  `'user'` and `'assistant'` (and rejects no third value structurally).
- `test_chat_messages_fk_cascade` — `ChatMessage.__table__.c.article_id`
  has a `ForeignKey` to `articles.id` with `ondelete == "CASCADE"` and
  `nullable is False`.
- `test_chat_messages_no_user_id` — `"user_id"` not in
  `ChatMessage.__table__.c` (ADR-11 / spec `chat_messages` #6).
- `test_articles_no_fake_columns` — none of `fake_title`,
  `fake_description`, `model`, `temperature`, `prompt_version`,
  `prompt_hash` are in `Article.__table__.c` (spec `articles` #2 and
  ADR-5).
- `test_articles_indexes` — `Article.__table__.indexes` contains
  exactly `ix_articles_source` and `ix_articles_published_at`. The
  `content_hash` unique constraint is **not** an additional non-unique
  index (spec `articles` #3).

### `tests/models/test_initial_migration_module.py`

These tests load the migration as a Python module via `importlib`
(globbing `backend/migrations/versions/*.py`, excluding `__pycache__` and
non-revision files) and assert:

- `test_single_revision_present` — exactly one `.py` revision file under
  `migrations/versions/` (excluding `.gitkeep`).
- `test_down_revision_is_none` — `module.down_revision is None`
  (genesis migration; spec "applies and produces the expected schema" #1).
- `test_revision_id_is_string` — `isinstance(module.revision, str)` and
  non-empty.
- `test_migration_imports_source_enum` — read the migration source as
  text and assert it contains the literal `from app.sources import Source`.
  This is the structural ADR-16 drift-prevention check from spec
  "Source StrEnum and FEED_URLS" #5.
- `test_migration_does_not_hardcode_enum_labels` — assert the migration
  source does **not** contain a duplicated string-list literal of the
  three labels (e.g. searches for `"NYT", "NPR", "Guardian"` as a single
  contiguous substring should not appear). The labels must be derived
  from `Source` at migration time.
- `test_upgrade_and_downgrade_callable` — `module.upgrade` and
  `module.downgrade` are callable. (Do not call them — they require an
  Alembic context and a live DB; that is QA territory.)

The schema-effect criteria from the spec that require a real Postgres —
enum violation rejected on insert, unique constraints rejected, FK
rejected, cascade delete propagates, `alembic upgrade head /
downgrade base / upgrade head` round-trip clean — are deliberately **not**
covered here. They are the QA agent's integration tests for 1.1.

## Open questions / contradictions surfaced

None that block `/start-dev`. Two adjacent observations the implementer
should be aware of but that are **out of scope** for 1.1 — flagged here so
they are not silently absorbed:

1. **Stale env-var fields in `app/config.py`.** The current
   [backend/app/config.py](../../backend/app/config.py) declares
   `prompt_version_transform`, `rss_feeds`, and `llm_cache_ttl_seconds`.
   These contradict the post-update `contracts.md` (Environment Variables
   table) and the rationale in ADR-5 (no `prompt_version`), ADR-9 (Redis
   is broker only — no LLM cache), and ADR-16 (no `RSS_FEEDS` env-var
   override). They are a leftover from iteration 0.C. Task 1.1 does not
   touch `config.py`; whichever later task next imports `settings`
   should propose a follow-up cleanup, or it can be folded into 1.3
   (scrape orchestrator) which is the first downstream consumer that
   actually reads source URLs.
2. **No `Article.fake` / `Article.messages` ORM relationships.**
   `architecture.md` describes the read API as a `LEFT JOIN` written at
   the SQL/select level, not via ORM relationship traversal, and no
   downstream Iter 1 task in the spec requires a back-ref. 1.1
   deliberately omits relationships; 1.5 (read API) decides whether to
   add them based on its own query shape.

No required changes to `contracts.md` were identified during planning.
