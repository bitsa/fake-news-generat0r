# Task 1.1 â€” Database Schema + First Migration (Spec)

## Source

Expands on **Iteration 1, Task 1.1 â€” Database Schema + First Migration** in
[plans/iteration-1.md](../../plans/iteration-1.md). Verbatim spec scope from that brief:

> - Tables created exactly as defined in `contracts.md`
> - Initial migration creates the `source_type` Postgres enum (`'NYT', 'NPR', 'Guardian'`)
>   and **three** application tables: `articles` (originals only), `article_fakes`
>   (1:1 with `articles`, all `NOT NULL`, PK = FK on `article_id` per ADR-5),

>   `article_versions` table (ADR-5).
> - Source identity lives as a Python `StrEnum` named `Source` in
>   `backend/app/sources.py`; the same module exports `FEED_URLS: dict[Source, str]`.
>   The Postgres enum is generated from `Source` so the two cannot drift.
> - Migration is hand-written, reversible (downgrade works â€” including
>   `DROP TYPE source_type`).

Authoritative shape definitions live in [contracts.md](../../contracts.md) (DB Schema)
and the rationale in [decisions.md](../../decisions.md) (ADR-3, ADR-5, ADR-16).

## Goal

Land the database foundation that every downstream Iteration 1 task assumes: a single
hand-written Alembic migration that creates the `source_type` Postgres enum and the
three application tables (`articles`, `article_fakes`, `chat_messages`) exactly as
defined in `contracts.md`, plus the matching SQLAlchemy ORM model classes
(`Article`, `ArticleFake`, `ChatMessage`) under `backend/app/models/`. Source
identity is fixed in a Python `StrEnum` (`Source`) so the Postgres enum and the
application's source list cannot drift. After this task, both the schema and the
ORM mapping are ready for the scrape orchestrator (1.3), transformation worker
(1.4), read API (1.5), and chat API (1.6) to plug in without further DDL or model
changes inside this iteration.

## User-facing behavior

- An operator running `alembic upgrade head` against an empty Postgres 16 database sees
  the migration apply cleanly and is left with a database containing the
  `source_type` enum type and the three application tables defined in `contracts.md`.
- An operator running `alembic downgrade base` afterwards is left with an empty
  database â€” no residual tables, indexes, or enum types from this migration.
- The schema visible to a `psql \d` listing matches `contracts.md` exactly: column
  names, types, nullability, defaults, primary keys, foreign keys, unique constraints,
  check constraints, and indexes.
- No HTTP surface area is added in this task; behavior is observable only at the
  database and Python-module level.

## Acceptance criteria

Each criterion is independently verifiable from outside the implementation. QA can
exercise them via `psql`, `information_schema` / `pg_catalog` queries, raw SQL
`INSERT`s, and importing `backend/app/sources.py` directly.

### Migration applies and produces the expected schema

1. After `alembic upgrade head` against an empty database, the migration completes
   without error and sets the alembic version to a single revision (no chain â€” this
   is the genesis migration).
2. After upgrade, the `source_type` Postgres enum exists with values exactly
   `('NYT', 'NPR', 'Guardian')` in that declaration order
   (verifiable via `pg_type` / `pg_enum` queries).
3. After upgrade, exactly three application tables exist: `articles`,
   `article_fakes`, `chat_messages`. No `sources` table. No `article_versions`
   table. No `article_embeddings` table.

### `articles` shape matches `contracts.md`

1. `articles` columns and types:
   `id SERIAL PRIMARY KEY`, `source source_type NOT NULL`, `title TEXT NOT NULL`,
   `description TEXT NOT NULL`, `url TEXT NOT NULL UNIQUE`,
   `published_at TIMESTAMPTZ` (nullable), `content_hash CHAR(64) NOT NULL UNIQUE`,
   `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
2. `articles` carries **no** fake-side columns â€” none of `fake_title`,
   `fake_description`, `model`, `temperature`, `prompt_version`, or `prompt_hash`
   are present.
3. `articles` has indexes `ix_articles_source` on `source` and
   `ix_articles_published_at` on `published_at DESC NULLS LAST`. The unique
   constraint on `content_hash` provides the dedup lookup index; **no** additional
   non-unique index on `content_hash` is created.
4. Attempting `INSERT INTO articles (source, ...) VALUES ('BBC', ...)` (any value
   outside the `source_type` enum) is rejected by Postgres with an
   invalid-text-representation / enum error.
5. Attempting to `INSERT` two `articles` rows with the same `url` is rejected by
   the unique constraint. Same for two rows with the same `content_hash`.

### `article_fakes` shape matches `contracts.md` (1:1 PK = FK per ADR-5)

1. `article_fakes` columns and types:
   `article_id INTEGER PRIMARY KEY REFERENCES articles(id) ON DELETE CASCADE`,
   `fake_title TEXT NOT NULL`, `fake_description TEXT NOT NULL`,
   `model VARCHAR(100) NOT NULL`, `temperature DOUBLE PRECISION NOT NULL`,
   `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
2. `article_fakes.article_id` is **simultaneously** the primary key and a foreign
   key to `articles(id)` â€” verifiable by inspecting `pg_constraint` (one PK
   constraint and one FK constraint, both on `article_id`).
3. The foreign-key delete action is `CASCADE` (verifiable in
   `information_schema.referential_constraints.delete_rule = 'CASCADE'`).
4. After inserting one article and one matching `article_fakes` row, attempting to
   `INSERT` a second `article_fakes` row with the same `article_id` is rejected by
   the primary-key violation.
5. After inserting one article and one matching `article_fakes` row,
   `DELETE FROM articles WHERE id = <that id>` removes the corresponding
   `article_fakes` row automatically (cascade).
6. Inserting an `article_fakes` row with `article_id` not present in `articles` is
   rejected by the foreign-key constraint.
7. `article_fakes` has no additional indexes beyond the implicit one on the
   primary key.

### `chat_messages` shape matches `contracts.md`

1. `chat_messages` columns and types:
   `id SERIAL PRIMARY KEY`,
   `article_id INTEGER NOT NULL REFERENCES articles(id) ON DELETE CASCADE`,
   `role VARCHAR(20) NOT NULL` with a `CHECK (role IN ('user', 'assistant'))`
   constraint, `content TEXT NOT NULL`,
   `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
2. `chat_messages` has index `ix_chat_messages_article_id` on `article_id`.
3. The foreign-key delete action on `chat_messages.article_id` is `CASCADE`.
4. Inserting a `chat_messages` row with `role = 'system'` (or any value other than
   `'user'` / `'assistant'`) is rejected by the CHECK constraint.
5. Deleting an `articles` row removes all `chat_messages` rows referencing it.
6. The `chat_messages` table has no `user_id` column (per ADR-11; auth is deferred).

### SQLAlchemy ORM models

1. The Python package `backend/app/models/` is importable as `app.models` and
   exposes three model classes:
   - `Article` (from `app.models.article` and/or re-exported on `app.models`)
   - `ArticleFake` (from `app.models.article_fake`)
   - `ChatMessage` (from `app.models.chat_message`)
2. Each model's `__tablename__` matches the migrated table:
   - `Article.__tablename__ == "articles"`
   - `ArticleFake.__tablename__ == "article_fakes"`
   - `ChatMessage.__tablename__ == "chat_messages"`
3. `Article.source` is mapped as `Mapped[Source]` (the same `Source` `StrEnum`
   from `app.sources`) backed by a SQLAlchemy column declared with
   `sa.Enum(Source, name="source_type", values_callable=...)` per the
   `contracts.md` snippet â€” i.e. the ORM-level enum mapping references the
   `Source` `StrEnum` directly, not a duplicated string list.
4. With the migration applied (`alembic upgrade head`), an end-to-end ORM
   round-trip via an `AsyncSession` succeeds:
   - Insert an `Article(source=Source.NYT, ...)` â†’ flush â†’ primary key populated.
   - Insert an `ArticleFake(article_id=<that id>, ...)` â†’ flush â†’ row persists.
   - Read each back via `session.get(...)` and confirm the persisted column
     values equal what was written.
   - Delete the `Article` â†’ the `ArticleFake` row is removed (cascade behavior
     observed through the ORM, not via raw SQL).
5. Constructing `Article(source="BBC")` (any value not in `Source`) is rejected
   â€” either at the Python typing layer (mypy / runtime `StrEnum` coercion) or
   at INSERT time by Postgres (`source_type` enum violation).
6. `ChatMessage.role` accepts `"user"` and `"assistant"` and is rejected for any
   other value at INSERT (the underlying CHECK constraint surfaces as an
   `IntegrityError` through the ORM).

### `Source` `StrEnum` and `FEED_URLS` (Python module)

1. `backend/app/sources.py` exists and is importable as `app.sources` from the
   backend package root.
2. The module exports a class `Source` that is a `StrEnum` with exactly three
   members: `NYT = "NYT"`, `NPR = "NPR"`, `GUARDIAN = "Guardian"` â€” and no others.
3. The module exports `FEED_URLS` annotated as `dict[Source, str]` with exactly
   three entries, one per `Source` member, each mapping to a non-empty RSS URL
   string. URLs match those listed in `contracts.md`:
   `https://rss.nytimes.com/services/xml/rss/nyt/HomePage.xml`,
   `https://feeds.npr.org/1001/rss.xml`,
   `https://www.theguardian.com/world/rss`.
4. Every key in `FEED_URLS` is a member of `Source` (i.e. `set(FEED_URLS) ==
   set(Source)`). No string keys; no extra entries.
5. The Postgres `source_type` enum's label set equals `{m.value for m in Source}`
   â€” i.e. the migration imports `Source` to derive labels, so adding a member to
   `Source` would change what the migration creates without a separate manual edit
   to a string list. (Verified by reading the migration source: it must reference
   `Source` rather than hardcoding `('NYT', 'NPR', 'Guardian')` as a duplicated
   string literal. This is the structural drift-prevention property required by
   ADR-16.)

### Reversibility of the migration

1. Running `alembic downgrade base` after `alembic upgrade head` removes all three
   application tables (`articles`, `article_fakes`, `chat_messages`) and runs
   `DROP TYPE source_type` â€” verifiable by re-querying `pg_type` and
   `information_schema.tables` afterwards.
2. After `alembic downgrade base`, the database state is indistinguishable from a
   fresh empty Postgres instance with respect to objects this migration created
   (no leftover types, tables, indexes, sequences, or constraints).
3. `alembic upgrade head; alembic downgrade base; alembic upgrade head` in
   sequence completes without error â€” the migration is idempotently re-applicable
   after a clean downgrade.

## Out of scope

- pgvector extension and the `article_embeddings` table â€” deferred to **Iteration 3
  task 3.2** (similarity detection). See [plans/iteration-3.md](../../plans/iteration-3.md).
- A `sources` DB table or any `enabled` toggle / runtime CRUD for sources â€”
  **deliberately rejected** per ADR-16. Source identity is config-only.
- An `article_versions` table or any 1:N fake-side relationship â€” **deliberately
  rejected** per ADR-5. Multi-prompt A/B is **Iteration 3 stretch task 3.6**, which
  documents its own migration to relax the 1:1 PK=FK constraint when (and only when)
  prompt versioning ships.
- `prompt_version` / `prompt_hash` columns on either table â€” **deliberately
  removed** for MVP per ADR-5.
- Seeding any rows into the new tables â€” the migration creates structure only.
- Any read or write API on top of the schema â€” covered by tasks 1.3, 1.5, and 1.6.
- An LLM-response-cache schema or Redis cache key namespace â€” **deliberately not
  added** per ADR-9 (Redis is the ARQ broker only).
- The Docker Compose service definitions, `alembic.ini`, or `migrations/env.py`
  bootstrap â€” those are **Iteration 0** deliverables and are assumed to already be
  in place before this task starts. This task adds a single revision file under
  `backend/migrations/versions/` and the `backend/app/sources.py` module.

## Open questions / assumptions

1. **Genesis revision â€” confirmed.** `backend/migrations/versions/` contains only
   `.gitkeep` (verified during spec review). The 0.C bootstrap test
   `test_T_E3_versions_dir_present` pins the directory's existence but expects no
   revisions yet. The migration's `down_revision` is `None`.
2. **`migrations/env.py` import wiring â€” confirmed in place.** `backend/alembic.ini`
   sets `prepend_sys_path = .`, which prepends the directory containing
   `alembic.ini` (i.e. `backend/`) to `sys.path` when alembic runs. From `backend/`,
   `app/sources.py` is importable as `app.sources` â€” both inside the container
   (cwd `/app`, `Makefile` target `backend-alembic`) and locally with the venv. No
   additional wiring required in 1.1.
3. **ORM model scope â€” resolved: in.** 1.1 lands the SQLAlchemy ORM model classes
   (`Article`, `ArticleFake`, `ChatMessage`) under `backend/app/models/` alongside
   the migration. Rationale: (a) `architecture.md`'s canonical layout already
   lists those three files; (b) `contracts.md`'s
   `source: Mapped[Source] = mapped_column(sa.Enum(Source, ...))` snippet has to
   live somewhere, and 1.1 is the natural home; (c) co-locating DDL and ORM
   definitions in one task prevents drift between them; (d) every downstream Iter 1
   task (1.3 / 1.4 / 1.5 / 1.6) needs the models, so deferring them just creates
   a hidden prerequisite for 1.3. ORM acceptance criteria are listed in the
   "SQLAlchemy ORM models" section above.
4. **`source_type` label declaration order.** `Source` declares members in the
   order `NYT`, `NPR`, `GUARDIAN`. The Postgres enum's label order is derived from
   the Python enum's iteration order. No acceptance criterion in this spec depends
   on enum ordinal (no `ORDER BY source` is asserted), but downstream consumers
   should not assume a different ordering.
5. **Test database isolation.** QA needs an empty Postgres 16 instance (the
   `pgvector/pgvector:pg16` image per ADR-12 is fine â€” the extension is not
   enabled in this task). How that is provisioned (docker-compose service,
   testcontainer, ephemeral schema) is a QA-plan concern, not a spec concern. The
   spec asserts behavior **on an empty database**, not on a particular harness.
6. **Index expression on `published_at`.** `contracts.md` specifies
   `CREATE INDEX ix_articles_published_at ON articles(published_at DESC NULLS LAST)`.
   Alembic's `op.create_index` supports this via expression / `postgresql_ops`.
   Assumed straightforward; surfaced here only because it is one of the few places
   the migration must use a non-default index expression rather than a column-name
   shortcut.
