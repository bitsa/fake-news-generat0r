You are auditing the work just completed on the current branch against this
project's five canonical specification documents. These documents are the
single source of truth for the system; the code on this branch must conform
to them, or the divergence must be deliberate and surfaced.

The five documents (in repo root) are:

1. architecture.md   — components, data flows, Docker Compose services,
                       directory structures.
2. contracts.md      — DB schema, REST endpoints (request/response shapes),
                       TypeScript types, env vars.
3. conventions.md    — Python + TypeScript style, logging rules, commit
                       format, branching, testing, definition of done.
4. decisions.md      — ADRs with rationale and rejected alternatives.
5. future_work.md    — explicitly deferred items + Loom talking track.

Your task — DO NOT MODIFY ANY FILE. This is a read-only audit.

Step 1 — Establish what changed on this branch.

- Run: git diff --stat $(git merge-base HEAD main)..HEAD
    (fall back to origin/main, master, or origin/master if main is missing)
- Run: git log --oneline $(git merge-base HEAD main)..HEAD
- Read each changed file fully. Do not skim.

Step 2 — Read all five spec documents fully:
  architecture.md, contracts.md, conventions.md, decisions.md, future_work.md.

Step 3 — For each changed file, check alignment against every spec doc that
could plausibly govern it. Specifically look for:

  contracts.md violations
    - DB columns, types, constraints, or table names that diverge from the
      schema in contracts.md
    - REST endpoint paths, methods, request bodies, or response shapes that
      don't match
    - TypeScript types that drift from their backend counterparts
    - New env vars not listed in contracts.md, or listed env vars used
      differently than documented

  architecture.md violations
    - New components, services, or processes not on the diagram
    - Changed data flows (e.g., scrape pipeline, read feed, chat streaming)
    - New Docker Compose services, ports, or networks
    - Backend or frontend directory layout that contradicts the documented
      structure

  conventions.md violations
    - Missing type hints, sync I/O where async is required, use of `Session`
      instead of `AsyncSession`
    - TypeScript without strict mode, class components, inline styles
    - Logs that include full prompts or secrets (explicit prohibition)
    - Commit messages, branch names, or test layout off-format
    - Definition-of-done items skipped

  decisions.md violations
    - Code that contradicts an ADR (e.g., introducing Celery, autogenerated
      migrations, CSS modules instead of Tailwind v3, sync ORM, etc.)
    - New significant decisions made in code that are NOT recorded as an ADR

  future_work.md violations
    - Items that were supposed to be deferred but appear to be implemented
      anyway (scope creep)
    - Items that were silently deferred in this branch but are NOT listed
      under future_work.md

Step 4 — Produce a single report with this exact structure:

```md
## Branch Alignment Report

### Branch summary
- Branch: <name>
- Files changed: <count>
- Commits: <count>
- One-paragraph plain-English summary of what this branch does.

### Aligned (briefly)
Bullet list of the major areas where the implementation matches the specs.
Keep this short — one line per area. The point is to confirm coverage, not
to praise.

### Divergences (the important section)
For each divergence, one entry:
  - **What:** one sentence describing the divergence.
  - **Where:** file:line references.
  - **Spec violated:** which doc and which section.
  - **Severity:** blocking | should-fix | acceptable-if-intentional.
  - **Recommendation:** fix the code, or update the spec doc, or flag
    to the user for a decision.

### Undocumented decisions
List any non-trivial implementation choices that should probably become
ADRs in decisions.md but currently aren't recorded anywhere.

### Suggested updates to spec docs
If reality has legitimately moved past the docs, list the specific edits
that the *update* skill should make. (Do not make them yourself.)
```

Rules:

- Cite file:line for every claim. No vague "somewhere in the backend".
- Do not edit any file. Do not run formatters or fixers.
- If the branch has no diff against the base, say so and stop.
- If a spec doc itself looks internally inconsistent with another, flag it
  in "Suggested updates" rather than picking a winner.
- Be terse. This report is read by a human who will then decide what to do.
